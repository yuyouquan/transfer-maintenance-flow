## 2. 项目概述

### 2.1 项目背景

在研项目完成开发后需要将维护职责从在研团队转交给维护团队，此过程涉及大量材料交付、评审审核和信息变更。当前流程依赖线下文档流转和人工跟踪，存在以下问题：

- 转维材料交付不规范，遗漏频繁
- 审核进度缺乏可视化，难以追踪
- 多角色并行协作效率低，信息传递依赖人工
- 缺乏AI辅助检查能力，质量把关依赖经验
- 历史记录和责任追溯困难

### 2.2 项目目标

1. **流程数字化**：将转维流程线上化，实现项目发起、资料录入、AI智能检查、维护审核、信息变更的全流程闭环管理
2. **效率提升**：通过多角色并行执行、AI智能检查、飞书通知等机制，将转维周期缩短30%以上
3. **质量保障**：通过CheckList模板和评审要素模板的标准化管理，确保转维材料质量
4. **可追溯性**：完整记录操作历史和审核意见，支持全流程追溯

### 2.3 项目范围

| 范围 | 说明 |
|------|------|
| 包含 | 工作台模块（发起申请、记录管理、待办、详情、流程操作）、配置中心模块（模板管理、版本管理） |
| 不包含 | 用户权限管理系统、飞书通知服务实现、项目管理系统对接、售后系统对接 |

### 2.4 前期策略

**本期仅实现前端界面 + Mock数据**，不涉及后端服务开发。具体策略：

- 所有页面使用Mock数据进行渲染和交互演示
- API接口层预留标准化接口定义，后续对接后端时无缝切换
- 飞书通知、AI检查等外部依赖以状态模拟的方式展示
- 重点验证交互流程和界面布局的合理性

## 5. 详细功能描述

### 5.1 W-001：发起转维电子流申请

**用户故事**

> 作为在研SPM，我希望能在工作台发起转维电子流申请，填写项目信息和人员配置，以便启动转维流程。

**界面字段**

| 字段名 | 类型 | 是否必填 | 说明 |
|--------|------|----------|------|
| 项目选择 | 下拉单选（支持搜索） | 是 | 枚举值如：X6870_H1234(Android16)、X6768_H1234等，选择后自动带出在研团队 |
| 在研团队-SPM | 人员选择 | 是 | 选择项目后自动回填，可修改 |
| 在研团队-TPM | 人员选择 | 是 | 选择项目后自动回填，可修改 |
| 在研团队-SQA | 人员选择 | 是 | 选择项目后自动回填，可修改 |
| 在研团队-底软集成开发代表 | 人员选择 | 是 | 选择项目后自动回填，可修改 |
| 在研团队-系统集成开发代表 | 人员选择 | 是 | 选择项目后自动回填，可修改 |
| 维护团队-SPM | 人员选择 | 是 | 手动选择 |
| 维护团队-TPM | 人员选择 | 是 | 手动选择 |
| 维护团队-底软集成开发代表 | 人员选择 | 是 | 手动选择 |
| 维护团队-系统集成开发代表 | 人员选择 | 是 | 手动选择 |
| 计划评审日期 | 日期选择器 | 是 | 选择预计评审日期 |
| 备注 | 多行文本 | 否 | 补充说明 |
| 转维指南 | 只读展示区 | - | 展示转维流程指南、转维CheckList、转维评审要素（标题+图片，图片支持放大预览） |

**验收标准**

1. 点击"项目转维申请"按钮后跳转到申请单页面
2. 选择项目后自动带出在研团队人员
3. 所有必填字段为空时提交按钮不可用或给出提示
4. 点击确定后在工作台生成一条转维电子流记录
5. 转维指南中的图片点击可放大预览

**业务规则**

- BR-001：项目选择后自动回填在研团队人员，但允许修改
- BR-002：点击确定后自动触发飞书通知，通知资料录入责任人
- BR-003：创建时自动完成"项目发起"节点，直接扭转到"资料录入与AI检查"节点
- BR-004：创建时自动将配置中心的转维材料和评审要素模板填充初始化，根据模板中的角色配置回填对应的项目人员

### 5.2 W-002：转维电子流记录列表

**用户故事**

> 作为系统用户，我希望能在工作台看到所有转维电子流记录，并根据自己的角色看到对应的操作按钮。

**列表字段**

| 字段名 | 说明 |
|--------|------|
| 项目名 | 转维申请关联的项目名称 |
| 发起人 | 发起转维申请的SPM |
| 节点 | 当前流水线所在的主节点 |
| 节点状态 | 当前节点的执行状态 |
| 最后更新时间 | 最近一次操作的时间 |
| 备注 | 当流水线被取消时显示取消原因 |
| 操作 | 详情、录入、评审、关闭流水线（条件显示） |

**操作按钮显示规则**

| 按钮 | 显示条件 |
|------|----------|
| 详情 | 所有记录均显示 |
| 录入 | 条件一：登录用户为普通用户，且在该申请的资料录入节点中存在未扭转到维护审核（或被拒绝回退）的条目，且资料录入-责任人包含当前用户。条件二：登录用户为该申请的角色负责人，且该角色的转维材料和评审要素未全部扭转到维护审核（或被拒绝回退） |
| 评审 | 登录用户对应该流水线的角色，且该角色已通过资料录入与AI检查并提交审核 |
| 关闭流水线 | 条件一：无角色进入维护审核时，仅申请人可见。条件二：有角色进入维护审核时，仅SQA可见 |

**验收标准**

1. 列表支持按项目名、发起人等字段搜索
2. 列表支持分页，默认每页20条
3. 操作按钮严格按显示规则条件渲染
4. 取消的流水线在备注列显示取消原因

### 5.4 W-004：转维电子流详情页

**用户故事**

> 作为系统用户，我希望查看某条转维申请的完整详情，包括流水线进度、团队配置、各条目状态和操作记录。

**页面模块（从上到下）**

#### 5.4.1 取消原因提示

- 当申请单被取消时，在详情页最上方显示取消原因（红色提示条）

#### 5.4.2 项目流水线

- 节点：项目发起 → 资料录入与AI检查 → 维护审核 → 信息变更
- 颜色规则：未开始-灰色、进行中-蓝色、成功-绿色、不通过/被拒绝-红色
- "资料录入与AI检查"和"维护审核"节点下方各用四个小圆点代表SPM/测试/底软/系统四个角色的子状态
- 小圆点颜色：灰色-未开始、蓝色-进行中、绿色-完成、红色-被拒绝
- 小圆点悬浮显示角色名称和状态

**流水线状态扭转规则**

- 当有一个角色的资料录入与AI检查通过并进入维护审核，则"维护审核"主节点变为进行中
- 当所有角色的维护审核全部通过，"资料录入与AI检查"和"维护审核"节点一起变为成功

#### 5.4.3 项目基础信息

| 项目负责人 | 项目总负责人 |

| 转维启动时间 | 申请创建时间 |
| 转维截止时间 | 计划评审日期 |
| 项目状态 | 流水线当前状态 |
| 备注 | 申请时填写的备注 |

#### 5.4.4 团队信息

- 在研团队：SPM、TPM、SQA、底软集成开发代表、系统集成开发代表
- 维护团队：SPM、TPM、底软集成开发代表、系统集成开发代表

#### 5.4.5 转维CheckList

| 字段 | 说明 |

| 序号 | 条目序号 |

| 录入状态 | 未录入（灰色）、已录入（绿色） |
| AI检查状态 | 进行中（蓝色）、通过（绿色）、不通过（红色）点击弹出Modal |
| 维护审核状态 | 未检查（灰色）、通过（绿色）、不通过（红色）点击弹出Modal |

- 展示流水线过程中创建的所有遗留任务

**界面结构**

- 顶部：流水线五节点进度展示

**录入操作**

1. 点击"录入"按钮弹出资料录入Modal
2. 填写资料录入内容（多行文本）
3. 两个按钮：暂存、确认
   - 暂存/关闭Modal：录入状态变为"已暂存（橙色）"
   - 确认：录入状态变为"已录入（绿色）"，触发AI检查（仅转维材料）

**委派操作**

1. 单条委派：点击条目后的"委派"按钮，弹出人员多选Modal
2. 批量委派：多选表格条目后，筛选右侧出现"全部委派"按钮
3. 确认委派后：被委派人员收到飞书通知，资料录入-责任人追加委派人员

**提交审核**

- 条件：该角色下所有转维材料和评审要素的录入状态均为"已录入"，且AI检查状态均为"通过"
- 操作：角色负责人点击"提交审核"按钮，弹出二次确认框
- 结果：该角色的数据扭转到维护审核节点，维护审核状态变为"审核中"

**业务规则**

- BR-005：暂存的资料可被多次修改，直到点击确认
- BR-006：确认后触发AI智能检查（转维材料），评审要素确认后不触发AI检查
- BR-007：当AI检查成功、进行中或失败时，重新录入会重新触发AI检查
- BR-008：委派操作为追加模式，不会替换原有责任人
- BR-009：只有角色负责人有权提交审核
- BR-010：被拒绝回退后，需重新录入并通过AI检查后才能再次提交审核

**验收标准**

1. TAB切换转维材料和评审要素正常
2. 数据范围按权限正确过滤
3. 录入Modal的暂存和确认按钮状态更新正确
4. 委派功能正常，支持批量操作
5. 提交审核按钮仅在条件满足时显示
6. 表格支持搜索和分页

7. 飞书通知对应角色负责人重新录入，同时通知SQA查看

**单条目审核**

- 支持多选后批量通过/不通过

**验收标准**

1. 固定栏不随页面滚动
2. 委派协作人功能正常
3. 通过流程中遗留任务创建正确

## 8. 非功能需求

### 8.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面首屏加载时间 | 不超过2秒 |
| 列表接口响应时间 | 不超过1秒 |
| 表格支持数据量 | 单页最多100条无卡顿 |
| 并发用户数 | 支持100用户同时在线 |

### 8.2 兼容性

| 项目 | 要求 |
|------|------|
| 浏览器 | Chrome 90+、Edge 90+、Firefox 90+、Safari 15+ |
| 分辨率 | 最低支持1280x720，适配1920x1080 |
| 设备 | 桌面端优先，暂不考虑移动端 |

### 8.3 可用性

- 所有操作提供明确的反馈（成功/失败提示）
- 关键操作（提交审核、关闭流水线）提供二次确认
- 表单页面支持草稿暂存
- 列表页面支持搜索和分页，默认按更新时间倒序

### 8.4 安全性

- 操作按钮根据用户角色和权限条件渲染
- 数据展示根据用户身份过滤
- 敏感操作需要二次确认

### 8.5 可维护性

- 前端代码模块化，按功能模块拆分
- 组件复用度高，表格、Modal、状态标签等抽取为通用组件
- Mock数据层与业务层解耦，后续对接后端时仅需替换数据层

### 附录A：枚举值定义

**流水线节点（PipelineNode）**
- PROJECT_INIT：项目发起
- DATA_INPUT_AI_CHECK：资料录入与AI检查
- MAINTENANCE_REVIEW：维护审核
- INFO_CHANGE：信息变更

**节点状态（NodeStatus）**
- NOT_STARTED：未开始（灰色）
- IN_PROGRESS：进行中（蓝色）
- SUCCESS：成功（绿色）
- REJECTED：不通过/被拒绝（红色）
- CANCELLED：已取消

**录入状态（InputStatus）**
- NOT_INPUT：未录入（灰色）
- DRAFT：已暂存（橙色）
- INPUTTED：已录入（绿色）

**AI检查状态（AICheckStatus）**
- NOT_STARTED：未开始（灰色）
- IN_PROGRESS：进行中（蓝色）
- PASSED：通过（绿色）
- FAILED：不通过（红色）

**维护审核状态（ReviewStatus）**
- NOT_REVIEWED：未审核（灰色）
- IN_REVIEW：审核中（蓝色）
- PASSED：通过（绿色）
- REJECTED：不通过（红色）

**角色类型（RoleType）**
- SPM
- TEST（测试/TPM）
- BSW（底软）
- SYS（系统）
