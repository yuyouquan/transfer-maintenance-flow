## 2. 技术架构

### 2.1 技术栈选型

| 类别 | 技术选型 | 版本 | 选型理由 |
|------|----------|------|----------|
| 框架 | Next.js 16 (App Router) | 16.x | 基于文件系统的路由、RSC、Layout嵌套、SSR/SSG支持 |
| 语言 | TypeScript | 5.x | 强类型保障，所有类型定义使用 `readonly` 确保不可变性 |
| UI组件库 | Ant Design | 5.x | 企业级后台组件库，表格/表单/Modal完善 |
| 样式方案 | TailwindCSS | 3.x | 原子化CSS，辅助布局与间距微调 |
| Ant Design SSR | @ant-design/nextjs-registry | - | 解决Ant Design在App Router下的SSR样式注入 |
| 包管理 | npm | - | 标准包管理工具 |

### 2.2 架构分层

```
┌─────────────────────────────────────────────────┐
│                   页面层 (Pages)                  │
│         App Router 路由页面组件                    │
├─────────────────────────────────────────────────┤
│                  组件层 (Components)              │
│    布局组件 / 流水线组件 / 状态标签 / 业务组件     │
├─────────────────────────────────────────────────┤
│                 服务层 (Services)                 │
│        API接口抽象层，当前指向Mock实现              │
├─────────────────────────────────────────────────┤
│                 Mock层 (Mock Data)               │
│        本地Mock数据，模拟后端API响应               │
├─────────────────────────────────────────────────┤
│                 类型层 (Types)                    │
│        TypeScript类型定义，全局共享                │
└─────────────────────────────────────────────────┘
```

### 2.3 核心设计原则

1. **不可变数据优先**：所有接口类型字段均使用 `readonly`，数组使用 `ReadonlyArray<T>`，状态更新通过创建新对象实现
2. **前后端分离**：业务组件通过Service层获取数据，Service层当前指向Mock，后续无缝切换为真实API
3. **组件化拆分**：按功能域拆分组件，单文件不超过400行，复杂文件不超过800行
4. **客户端组件最小化**：仅需要交互状态的组件声明 `'use client'`，纯展示组件尽量保持为Server Component

## 5. 数据模型设计

### 5.1 类型定义总览

所有类型定义集中在 `src/types/index.ts`，遵循以下原则：

- 所有接口字段使用 `readonly` 修饰符
- 所有数组使用 `ReadonlyArray<T>` 而非 `T[]`
- 枚举值使用联合类型（Union Type）而非 `enum` 关键字
- 可选字段使用 `?` 标注

### 5.2 核心枚举类型

```typescript
/** 流水线主节点状态 */
type PipelineNodeStatus = 'not_started' | 'in_progress' | 'success' | 'failed';

/** 角色子节点状态 */
type RoleNodeStatus = 'not_started' | 'in_progress' | 'completed' | 'rejected';

/** 录入状态 */
type EntryStatus = 'not_entered' | 'draft' | 'entered';

/** AI检查状态 */
type AICheckStatus = 'not_started' | 'in_progress' | 'passed' | 'failed';

/** 维护审核状态 */
type ReviewStatus = 'not_reviewed' | 'reviewing' | 'passed' | 'rejected';

/** 流水线整体状态 */
type PipelineStatus = 'in_progress' | 'completed' | 'cancelled';

/** 角色类型（全量） */
type RoleType = 'SPM' | 'TPM' | 'SQA' | '底软' | '系统';

/** 流水线并行角色（四角色） */
type PipelineRole = 'SPM' | '测试' | '底软' | '系统';

/** 团队类型 */
type TeamType = 'research' | 'maintenance';
```

### 5.3 核心数据模型

| 模型 | 接口名 | 用途 | 关键字段 |
|------|--------|------|----------|
| 团队成员 | `TeamMember` | 表示单个团队成员 | id, name, role, avatar, department |
| 项目团队 | `ProjectTeam` | 在研/维护两个团队 | research, maintenance |
| 项目 | `Project` | 项目基础信息 | id, name, code, team |
| 转维申请 | `TransferApplication` | 核心业务实体 | id, projectName, applicant, status, pipeline, team |
| 流水线状态 | `PipelineState` | 四节点 + 角色子状态 | projectInit, dataEntry, maintenanceReview, infoChange, roleProgress |
| 角色进度 | `RoleProgress` | 单角色在流水线中的状态 | role, entryStatus, reviewStatus |
| 转维材料 | `CheckListItem` | 单条CheckList | seq, checkItem, entryStatus, aiCheckStatus, reviewStatus, deliverables |
| 评审要素 | `ReviewElement` | 单条评审要素 | seq, standard, description, entryStatus, aiCheckStatus, reviewStatus |
| 交付件 | `Deliverable` | 文件/链接 | name, url, type |
| Block任务 | `BlockTask` | 审核不通过时创建 | description, resolution, responsiblePerson, deadline, status |
| 遗留任务 | `LegacyTask` | 审核通过时可选创建 | responsiblePerson, description, deadline, status |
| 历史记录 | `HistoryRecord` | 操作日志 | action, operator, detail, timestamp |
| 待办任务 | `TodoItem` | 待办入口 | projectName, node, type |
| CheckList模板 | `CheckListTemplate` | 转维材料模板 | type, checkItem, responsibleRole, entryRole, reviewRole |
| 评审要素模板 | `ReviewElementTemplate` | 评审要素模板 | standard, description, responsibleRole |
| 模板版本 | `TemplateVersion` | 模板版本管理 | version, type, createdAt, createdBy, itemCount |
| 关闭评审行 | `CloseReviewRow` | 关闭流水线时评审状态 | role, conclusion, comment |
| 查询参数 | `ApplicationListParams` | 列表查询 | keyword, page, pageSize |
| 分页结果 | `PaginatedResult<T>` | 通用分页响应 | data, total, page, pageSize |

### 5.4 模型关系图

```
Project (1) ────── (N) TransferApplication
                         │
                         ├── (1) PipelineState

                         ├── (1) ProjectTeam
```

## 6. 组件设计

### 6.1 布局组件

#### AppLayout（全局布局）

- **文件**：`src/components/layout/AppLayout.tsx`
- **类型**：客户端组件（`'use client'`）
- **职责**：提供全局侧边栏导航 + 顶部工具栏
- **结构**：

```
┌──────────────────────────────────────────────┐
│ Sider(可折叠)  │  Header (用户信息、通知图标)  │
│ ┌────────────┐ │  ─────────────────────────── │
│ │ 转维流程系统│ │                              │
│ │            │ │                              │
│ │ 转维项目   │ │       Content                │
│ │ 设置       │ │       (children)             │
│ │            │ │                              │
│ └────────────┘ │                              │
└──────────────────────────────────────────────┘
```

- **状态**：`collapsed: boolean`（侧边栏折叠状态）
- **导航项**：
  - `/workbench` — 转维项目
  - `/config` — 设置
- **依赖**：Ant Design Layout/Menu/Avatar/Dropdown

### 6.2 流水线组件

#### PipelineProgress（流水线进度展示）

- **文件**：`src/components/pipeline/PipelineProgress.tsx`
- **类型**：客户端组件
- **Props**：

| Prop | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| pipeline | `PipelineState` | 必填 | 流水线完整状态数据 |
| showRoleDots | `boolean` | `true` | 是否展示角色子状态小圆点 |

- **颜色映射规则**：

| 主节点状态 | 颜色 |

|-----------|------|
| not_started（未开始） | 灰色 `#d9d9d9` |
| in_progress（进行中） | 蓝色 `#1677ff` |
| completed（已完成） | 绿色 `#52c41a` |

## 8. 状态管理方案

### 8.1 方案选型

采用 **React useState + 组件内状态** 方案，不引入额外状态管理库。

**选型理由**：
- 本系统以表单和列表展示为主，状态复杂度适中
- 页面间数据独立性强，无需全局共享状态
- 减少外部依赖，降低项目复杂度
- 后续如需求升级，可平滑迁移到Zustand或Context

### 8.2 状态分层

| 层级 | 管理方式 | 典型场景 |
|------|----------|----------|
| 页面级状态 | `useState` | 表格数据、分页参数、搜索关键字、Tab选中项 |
| 组件级状态 | `useState` | Modal可见性、表单输入、loading状态 |
| 衍生状态 | `useMemo` | 按条件过滤的列表、按钮显示条件计算 |
| 副作用 | `useEffect` | 页面加载数据、状态变更后触发通知 |

### 8.3 状态更新规范

所有状态更新必须遵循不可变原则：

```typescript
// 正确：创建新对象
const handleUpdate = (id: string, field: string, value: string) => {
  setItems(prev =>
    prev.map(item =>
      item.id === id ? { ...item, [field]: value } : item
    )
  );
};

// 错误：直接修改
// item.field = value; // 禁止
```

```typescript
// 正确：创建新数组
const handleAdd = (newItem: CheckListItem) => {
  setItems(prev => [...prev, newItem]);
};

// 正确：过滤删除
const handleRemove = (id: string) => {
  setItems(prev => prev.filter(item => item.id !== id));
};
```

### 8.4 数据获取模式

```typescript
// 页面组件中的数据获取模式
'use client';

export default function WorkbenchPage() {
  const [applications, setApplications] = useState<ReadonlyArray<TransferApplication>>([]);
  const [loading, setLoading] = useState(false);
  const [params, setParams] = useState<ApplicationListParams>({
    page: 1,
    pageSize: 20,
  });

  useEffect(() => {
    const fetchData = async () => {
      setLoading(true);
      try {
        const result = await applicationService.getList(params);
        setApplications(result.data);
      } catch (error) {
        message.error('数据加载失败');
      } finally {
        setLoading(false);
      }
    };
    fetchData();
  }, [params]);
}
```

## 10. 前后端分离策略

### 10.1 分层架构

```
页面组件 (Page)
    │ 调用
    ▼
Service层 (services/)
    │ 当前实现         后续实现
    ├──────────┐    ┌──────────┐
    ▼          ▼    ▼          ▼
Mock层       →→→  fetch API
(mock/)            (后端REST API)
```

### 10.2 切换策略

当后端API就绪时，仅需修改Service层实现：

| 步骤 | 操作 | 影响范围 |
|------|------|----------|
| 1 | 在Service文件中将Mock引用替换为fetch调用 | 仅 `services/*.ts` 文件 |
| 2 | 配置API基础URL（环境变量） | `.env` 文件 |
| 3 | 添加请求拦截器（Token注入、错误处理） | `lib/request.ts` |
| 4 | 移除或保留Mock层（可保留用于开发/测试） | 无代码变更 |

### 10.3 环境变量预留


### 10.4 请求工具预留

```typescript
// lib/request.ts（后续对接时创建）

const BASE_URL = process.env.NEXT_PUBLIC_API_BASE_URL || '';

export async function request<T>(path: string, options?: RequestInit): Promise<T> {
  const response = await fetch(`${BASE_URL}${path}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      // Authorization: `Bearer ${token}`,  // 后续添加
      ...options?.headers,
    },
  });

  if (!response.ok) {
    throw new Error(`请求失败: ${response.status}`);
  }

  return response.json();
}
```

## 11. 编码规范

### 11.1 不可变数据规范

**核心原则：始终创建新对象，永不修改已有对象。**

#### 类型定义

```typescript
// 正确：所有字段使用 readonly
interface CheckListItem {
  readonly id: string;
  readonly seq: number;
  readonly entryStatus: EntryStatus;
  readonly deliverables: ReadonlyArray<Deliverable>;
}

// 错误：可变字段
interface CheckListItem {
  id: string;         // 缺少 readonly
  items: Deliverable[]; // 应使用 ReadonlyArray
}
```

#### 状态更新

```typescript
// 正确：展开操作符创建新对象
const updatedItem = { ...item, entryStatus: 'entered' as const };

// 正确：map创建新数组
const updatedList = items.map(i => i.id === id ? { ...i, status: newStatus } : i);

// 正确：filter创建新数组
const filteredList = items.filter(i => i.id !== removedId);

// 错误：直接修改
item.entryStatus = 'entered';   // 禁止
items.push(newItem);             // 禁止
items[0].status = 'new';        // 禁止
```

### 11.2 文件组织规范

| 规则 | 要求 |
|------|------|
| 单文件行数 | 典型200-400行，最大不超过800行 |
| 函数体行数 | 不超过50行，超出则提取子函数 |
| 嵌套深度 | 不超过4层，使用早返回（early return）简化 |
| 组件粒度 | 一个文件一个组件，按功能域分目录 |
| 导出方式 | 优先 `export default`（页面组件），`export` 用于工具函数 |

### 11.3 命名规范

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 组件文件 | PascalCase | `PipelineProgress.tsx`、`StatusTag.tsx` |
| 工具文件 | camelCase | `constants.ts`、`utils.ts` |
| 页面文件 | 固定名称 | `page.tsx`、`layout.tsx` |
| 组件名 | PascalCase | `AppLayout`、`CheckListTable` |
| 函数名 | camelCase | `handleSubmit`、`getApplicationList` |
| 常量 | UPPER_SNAKE_CASE | `STATUS_COLORS`、`NODE_LABELS`、`MOCK_USERS` |

### 11.4 组件编写规范

```typescript
// 标准组件结构
'use client'; // 仅需要交互时添加

import React, { useState, useMemo } from 'react';
import { Table, Button } from 'antd';
import type { CheckListItem } from '@/types';

// Props接口声明，使用 readonly
interface CheckListTableProps {
  readonly items: ReadonlyArray<CheckListItem>;
  readonly onEntryClick: (id: string) => void;
  readonly loading?: boolean;
}

// 常量提取到组件外部
const PAGE_SIZE = 20;

// 组件默认导出
export default function CheckListTable({
  items,
  onEntryClick,
  loading = false,
}: CheckListTableProps) {
  const [currentPage, setCurrentPage] = useState(1);

  // 衍生状态使用useMemo
  const paginatedItems = useMemo(
    () => items.slice((currentPage - 1) * PAGE_SIZE, currentPage * PAGE_SIZE),
    [items, currentPage]
  );

  return (
    <Table
      dataSource={[...paginatedItems]}
      loading={loading}
      rowKey="id"
      pagination={{
        current: currentPage,
        pageSize: PAGE_SIZE,
        total: items.length,
        onChange: setCurrentPage,
      }}
    />
  );
}
```
